# PMO 系统判断文档

> 基于当前 `PMO_SYSTEM_BASELINE.md` 的第一轮判断，用来区分这套 PMO 体系里哪些已经是有效沉淀，哪些更像演化过程中留下来的草率决定或历史噪音。

## 目的

这份文档不是为了立即重构 PMO。

它主要服务两个目标：

- 帮助你建立对当前 PMO 体系的判断力
- 为以后是否整理、收口、抽离提供判断依据

## 前提判断

当前这套 PMO 体系更像是“演化式形成”的系统，而不是“先完整设计、再严格落地”的系统。

也就是说：

- 它先是为了解决真实协作问题而长出来
- 后来才逐步补 manual、workflow、protocol、skills
- 很多 contract 是在使用中形成的，不是一开始就被正式设计出来的

所以现在最重要的不是批评它“不够优雅”，而是先区分：

- 哪些东西虽然不是一开始设计好的，但已经证明有价值
- 哪些东西只是历史阶段性的临时决策，后续可以收敛

## 一、当前看起来属于“有效沉淀”的部分

这些部分虽然未必一开始就设计得很完整，但现在已经有明显的系统价值。

### 1. 文件驱动的 PMO 状态机

这是当前体系里最有价值的沉淀之一。

具体表现：

- `discussion_batches`
- `idea_backlog`
- `sprint_candidates`
- `current_sprint`
- `execution_task`
- `execution_report`

这些文件已经共同形成了一套可见、可追踪、可回放的状态机。

它的价值在于：

- PMO 状态不只存在于对话里
- handoff 和 report 可以落盘
- 当前在做什么、下一步能做什么、为什么停在这里，都能被文件回答

这个结构未必最优雅，但已经明显不是“草率遗留”，而是有效运行面。

### 2. `current_sprint` 和 `execution_task` 的分工

这是一个非常重要的有效设计。

当前体系里已经形成了：

- `current_sprint.md` = 轻量 PMO 状态卡
- `execution_task.md` = 重 execution contract

这个分工很有价值，因为它避免了：

- PMO 状态和执行合同混成一个文件
- 当前状态太重，不便扫描
- handoff 被随手写在聊天里而失去稳定性

这部分应该视为有效沉淀。

### 3. Discussion -> Promotion -> Execution 的阶段拆分

这套阶段拆分已经很像真正的操作系统骨架了。

具体价值是：

- discussion 不会直接跳 execution
- execution-ready 和 discussion-stage 的东西不会混在一起
- durable decision 也有单独去处

这说明你们不是把所有主题都塞进 backlog，而是已经建立了层次感。

这也是有效沉淀。

### 4. Closeout 与 Commit 的分离

这是我认为很成熟的一点。

很多团队会默认：

- 做完 = close
- close = commit

但你们当前体系里已经把：

- delivery state
- validation state
- PMO closeout
- git commit

分开看待了。

这对于真实协作很重要，也说明这套系统已经不只是任务列表，而是有交付语义。

### 5. Public / Private Core 边界被纳入 PMO

这不是简单的“文档多写了一点”，而是一个很有价值的系统决策。

因为它意味着：

- PMO 不是脱离架构现实的
- escalation 是架构敏感的
- execution worker 的边界不是纯工程边界，而是产品/架构边界

这会让 PMO 比普通 sprint board 更有约束力。

## 二、当前看起来更像“草率遗留”的部分

这些部分不是完全没用，但更像是演化过程中临时长出来、现在已经开始带来理解成本的部分。

### 1. 多层文档之间的边界不够干净

当前 manual、workflow、protocol、guide、skill 之间有不少内容重叠。

典型问题：

- 有些规则在 manual 里写一次，在 workflow 里再写一次
- 有些规则既存在于 protocol，也存在于 skill wording
- 有些文档在说“规则”，有些文档在说“当前做法”，边界不完全稳定

这类问题不一定立刻致命，但它明显属于“演化堆叠”的痕迹。

### 2. Skill 对路径和项目现实的依赖过重

当前 `.codex/skills/**` 不只是引用 PMO，它已经部分内嵌了项目现实。

这说明：

- skill 不只是通用方法
- skill 也在承担项目耦合层

这在项目内部短期很好用，但从系统设计看，确实更像是“先跑起来”的临时收口方式，而不是很干净的架构。

### 3. 一些 contract 只存在于习惯里，不存在于单点定义里

比如：

- 什么叫 active handoff
- 什么时候应该 archive outbox
- 哪些情况算 closeout-ready 但 not committed

这些规则虽然已经在工作，但很多时候是分散在多个文件和使用习惯中的。

这就导致：

- 你如果不熟，得自己拼
- 一旦换人，很容易误判

这种情况通常就是演化式系统常见的“隐性 contract 过多”问题。

### 4. 角色命名和实际控制面的绑定过深

当前体系里 `Codex / Claude / Human` 的分工很明确，这本身没问题。

但问题在于：

- 某些地方写的是角色
- 某些地方写的是具体产品
- 某些地方两者混用了

这会让系统在理解上变得不够一致。

这类问题更像历史阶段性决定，而不是稳定设计。

### 5. “哪些文件是运行面，哪些文件只是解释面”没有完全被单独声明

你现在之所以会担心随便动 PMO 出后果，一个核心原因就是：

- 运行面和说明面虽然事实上已经不同
- 但没有在最早阶段被清晰制度化

所以现在需要靠基线文档反向帮你识别：

- 哪些文件碰了会影响系统行为
- 哪些文件只是帮助理解

这说明原设计阶段在这件事上确实没有完全想清楚。

## 三、当前最缺的不是“重构能力”，而是“判断层”

你现在最缺的不是动手改的能力，而是判断框架。

也就是：

- 什么必须保留
- 什么可以重写
- 什么先不动
- 什么以后再抽

如果没有这个判断层，过早提取或重构很容易犯一个问题：

把有效沉淀和历史噪音一起打包重写。

这样代价通常很大。

## 四、当前建议

基于现在的理解，我的建议是：

### 1. 暂时不要把 PMO 当成“待提取资产”

更适合先把它当成：

- 已经在工作的轻量控制面
- 但还没被彻底整理过的系统

### 2. 先继续补“判断文档”，不要急着补“抽离文档”

当前更值得做的是：

- 基线
- 判断
- 地图
- 风险点

而不是立即设计新的抽离结构。

### 3. 以后如果真的整理，优先级应该是：

1. 先收 contract
2. 再收边界
3. 再收文档分工
4. 最后才考虑提取或模板化

### 4. 现在最值得保守保护的部分

如果后面要做任何整理，当前最值得优先保护的是：

- PMO 状态机文件组
- `current_sprint` / `execution_task` / `execution_report` 分工
- discussion -> promotion -> execution 的阶段拆分
- closeout 与 commit 的分离
- public/private core 边界进入 PMO escalation 逻辑

## 五、当前一句话结论

当前 PMO 体系不是“设计成熟的架构系统”，而是“已经被真实使用磨出来、但还没有被系统整理过的控制面”。

所以现在更合理的方向不是贸然提取，而是先提升判断力，先把有效沉淀和草率遗留分开看。
